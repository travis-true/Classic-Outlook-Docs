## 2. Purpose

This plan defines how to build the final employee guide. The workplace performance problem is not a lack of general Outlook knowledge. Employees need quick, reliable help deciding what an email needs, making follow-up visible and completing common inbox tasks without losing track of work. The final guide must work as a task-based reference that employees can search, skim and use while Classic Outlook is open.

## 3. Performance outcomes

Employees who use the final guide should be able to:

1. Identify Classic Outlook and use the main mail layout.
2. Read, reply, reply all, forward, create and send messages.
3. Attach files or share cloud links using safe work-account habits.
4. Search for messages, people and attachments.
5. Decide whether an email needs action and choose the next visible follow-up step.
6. Use flags, reminders, categories, folders, archive, delete, Focused Inbox and rules.
7. Use email-related calendar follow-up, appointments and meetings.
8. Configure automatic replies and signatures.
9. Open and use shared mailboxes without sharing credentials.
10. Use Microsoft 365 Copilot for summaries, decisions, action items, draft replies, rewriting, coaching and meeting creation where available.
11. Review Copilot output before sending because **Copilot starts the draft. You finish the message.**
12. Troubleshoot missing commands or unavailable features and use approved help resources.

## 4. Core workflow

The guide is organized around this workflow:

**Email → Decide → Make visible → Follow through**

The three recurring questions are:

- **Does this need action from me?**
- **What is the next step?**
- **Where will I make it visible?**

These messages should appear in the start-here section, decision pages, follow-up tasks, shared-mailbox tasks, Copilot review tasks and final reminders. They should recur as short cues, not repeated long explanations.

## 5. Audience and use context

The audience includes employees with mixed Outlook experience, high email volume, shared-mailbox responsibilities and field-support needs. The guide must support fast task retrieval, keyboard-and-mouse workflows, managed Windows devices, work accounts only and accessible use while Outlook is open. It should avoid jargon, define terms and provide short success checks.

## 21. UDL and accessibility plan

Engagement: immediate relevance, clear purpose, predictable structure, low-friction task entry, self-checks and no unnecessary time pressure. Representation: plain language, text plus visuals, descriptive headings, real tables, alt text, icons with labels, consistent callouts, defined acronyms and no sensory-dependent instructions. Action and expression: keyboard and mouse support where useful, descriptive links, logical reading order, clear success checks, alternative support path and no color-only cues.

Accessibility target: WCAG 2.1 AA principles, screen-reader-compatible Word, minimum 11-point body text, 4.5:1 normal-text contrast, accessible tables, real heading styles, real lists, inline images, alt text, English (United States) and human accessibility testing. Do not claim compliance from automation alone.

## 22. Cognitive-load rules

Use one task per procedure, short step groups, progressive disclosure, chunked troubleshooting, limited callouts, consistent page patterns, clear prerequisites, minimal duplication, separation of what to do from why, advanced exceptions outside the basic flow and cross-references instead of repeated procedures.

## 23. BCBSKS Plain Talk

Use clear over clever, helpful over decorative, human and approachable wording, warm but professional tone, sixth- to eighth-grade reading level, active voice, common words, short sentences, one main idea per paragraph, descriptive headings, bullets and numbered steps and most important information first. Avoid justified text, dense paragraphs, unexplained acronyms, all-caps emphasis and decorative language.

## 13. Build sequence  
1. Approve the detailed build plan.  
2. Build the page-by-page copy plan.  
3. Draft the full written copy.  
4. Validate current Classic Outlook procedures.  
5. Capture screenshots using dummy data.  
6. Add Snagit annotations.  
7. Build the Word document.  
8. Complete accessibility checks.  
9. Conduct SME and user review.  
10. Publish the PDF to SharePoint.  
11. Add metadata and related-resource links.  
12. Record version and review date.  
This structure follows the project’s performance-support standards: task-based procedures, scan-first design, screenshot placeholders, troubleshooting, “What to remember,” “Avoid this,” accessibility and support guidance. The existing training materials also establish the core flow of deciding what an email needs, making follow-up visible and using Copilot with human review.
