## Platform-difference and master-guide integration map  
## Take Control of Your Inbox: Classic Outlook for Windows  
## 1. Purpose  
Use this map to:  
* Protect the accuracy of the Classic Outlook guide  
* Prevent instructions from different Outlook versions from being mixed  
* Prepare the Classic Outlook content for later combination into one master guide  
* Identify tasks that require significant rewriting on other platforms  
* Track features that are unavailable or configuration-dependent  
* Create consistent cross-platform labels and terminology  
This is an author-facing asset. It is not intended for publication as a participant guide in its current form.  
  
## 2. Guide-series structure  
The completed guide series will include:  
1. Classic Outlook for Windows  
2. New Outlook for Windows  
3. Outlook on the web  
4. Outlook on iPadOS  
5. Combined master guide, when approved  
Each guide should retain the same major task order:  
1. Get oriented  
2. Work with email  
3. Make follow-up visible  
4. Organize the inbox  
5. Search and find  
6. Configure Outlook  
7. Work with shared mailboxes  
8. Use Copilot  
9. Troubleshoot and get help  
This structure allows employees to locate the same task in any guide without learning a new organization system.  
  
## 3. Source-of-truth hierarchy  
Use this hierarchy when validating procedures:  
1. Approved BCBSKS workflows and terminology  
2. Current Microsoft Support documentation  
3. Validated screenshots from the managed environment  
4. Direct testing in the employee environment  
5. Existing guide content  
When a Microsoft procedure differs from the managed environment:  
* Document the employee experience  
* Add an availability note  
* Record the difference in the author notes  
* Validate with the appropriate technical owner  
* Avoid promising a feature that employees cannot access  
  
## 4. Standard platform labels  
Use these labels consistently in all four guides.  
## Same on most platforms  
Use when the task and expected result are substantially similar, even if a button moves.  
**Example:** Reply, Reply All and Forward use different screen locations, but their purpose remains the same.  
  
## Different in New Outlook  
Use when New Outlook changes:  
* Command location  
* Settings path  
* Terminology  
* Available options  
* Feature behavior  
  
## Different on the web  
Use when Outlook on the web depends on:  
* Browser menus  
* Web settings  
* Browser permissions  
* Separate browser windows  
* Web-only or limited features  
  
## Different on iPad  
Use when Outlook on iPadOS depends on:  
* Touch controls  
* Mobile menus  
* Swipe actions  
* Long-press actions  
* Apple Magic Keyboard shortcuts  
* Files or iPadOS sharing controls  
* Mobile feature limitations  
  
## Not available  
Use when a task or capability is not supported on the platform.  
Do not substitute a loosely related workflow without explaining the difference.  
  
## Availability may vary  
Use when the capability depends on:  
* Microsoft 365 licensing  
* Outlook version  
* Update channel  
* Mailbox type  
* Administrator settings  
* Jamf configuration  
* Connected experiences  
* Exchange Online support  
* Gradual Microsoft rollout

## 14. Common copy that should remain identical  
Reuse these approved concepts in every guide.  
## Core workflow  
**Email → Decide → Make visible → Follow through**  
## Decision questions  
1. Does this need action from me?  
2. What is the next step?  
3. Where will I make it visible?  
## Four doors  
* Respond now  
* Do the task  
* Make follow-up visible  
* Let it go  
## Follow-up principle  
Visible beats remembered.  
## System principle  
A simple system you use is better than a perfect system you ignore.  
## Copilot principle  
Copilot starts the draft. You finish the message.  
## Review categories  
* Accuracy  
* Recipients  
* Owners  
* Dates and deadlines  
* Tone  
* Completeness  
* Attachments and links  
* Sensitive details  
* Final wording  
## Shared mailbox principle  
Confirm the From address and ownership before sending.

## 16. Master-guide assembly options  
## Option A — Task-first comparison guide  
Recommended for the eventual master guide.  
## Example section  
**Task: Flag a message for follow-up**  
* Classic Outlook steps  
* New Outlook steps  
* Outlook on the web steps  
* Outlook on iPadOS steps  
* Comparison notes  
* Common reminder  
**Advantages:**  
* Best for employees switching devices  
* Makes differences easy to compare  
* Matches the original project goal  
* Reduces repeated conceptual guidance  
**Challenges:**  
* Can become wide and dense  
* Requires disciplined page layouts  
* Screenshots must be carefully limited  
  
## Option B — Four platform guides combined into one file  
## Structure  
* Part 1: Classic Outlook  
* Part 2: New Outlook  
* Part 3: Outlook on the web  
* Part 4: Outlook on iPadOS  
**Advantages:**  
* Simplest production method  
* Existing guides can be combined with minimal editing  
* Easier to maintain by platform  
**Challenges:**  
* Repeats common concepts  
* Harder to compare the same task  
* Creates a long document  
  
## Recommendation  
Build and publish the four separate guides first.  
After all guides are validated, create a shorter **task-first master comparison guide** rather than simply merging all pages.  
The master guide should:  
* Explain the common workflow once  
* Use concise procedures  
* Link to detailed platform guides  
* Focus on differences  
* Include a platform decision table  
* Avoid repeating full troubleshooting sections

## 20. Final author checklist  
Before approving a platform guide:  
* The correct platform is named on every page.  
* No screenshots from another platform appear.  
* Settings paths match the managed environment.  
* Shared mailbox behavior was tested.  
* Copilot was tested in the primary mailbox.  
* Unsupported features are labeled.  
* Configuration-dependent features use availability notes.  
* Common workflow language remains consistent.  
* Cross-platform differences are recorded.  
* The content ID is assigned.  
* Screenshot IDs match the task IDs.  
* The guide can be integrated into the future master guide.  
* Current Microsoft Support references were checked.  
* Last reviewed date is recorded.  
The map continues the project’s modular, workflow-focused performance-support approach and preserves consistent screenshot, accessibility and support conventions.
