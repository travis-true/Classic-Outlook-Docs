## Word production and publishing plan  
## Take Control of Your Inbox: Classic Outlook for Windows  
## 1. Purpose  
Use this plan to build the approved copy, screenshots and reference content into a polished Microsoft Word document that can be:  
* Reviewed by subject matter experts  
* Updated over time  
* Exported as an accessible PDF  
* Published in SharePoint  
* Combined later with other Outlook platform guides  
* Used as a searchable performance-support manual

## 3. Document setup  
## Page setup  
Use:  
* US Letter  
* Portrait orientation  
* 0.7-inch top and bottom margins  
* 0.75-inch left and right margins  
* Header distance: 0.3 inch  
* Footer distance: 0.3 inch  
* Vertical alignment: Top  
Use landscape pages only when a table cannot remain readable in portrait orientation.  
## Page numbering  
Use Arabic page numbers beginning with the About this guide page or the first content page, depending on the final design decision.  
Recommended:  
* Cover: No page number  
* Front matter: Roman numerals optional, but not necessary  
* Main guide: Arabic numbers  
* Use one continuous numbering system for easiest PDF navigation  
## Section breaks  
Insert section breaks:  
1. After the cover  
2. Before the table of contents when needed  
3. Before any landscape comparison table  
4. Before the final reference section when header or footer treatment changes  
Avoid unnecessary section breaks because they complicate headers, footers and page numbering.

* Hanging indent: 0.15 inch

* Centered or left-aligned consistently

* Dark gray

## 8. Procedure-page layout  
Use one of two standard layouts.  
## Layout A — Full-width procedure  
Best for:  
* Multi-step instructions  
* Tables  
* Troubleshooting content  
* Copilot review checklists  
* Shared mailbox procedures  
Structure:  
1. Task title  
2. Use this when  
3. Before you start  
4. Numbered steps  
5. Screenshot  
6. What happens next  
7. What to remember  
8. Avoid this  
9. Platform difference  
10. Related tasks  
## Layout B — Two-column procedure  
Best for:  
* Short procedures  
* Short copy beside one screenshot  
* Before-and-after examples  
* Decision tables  
Recommended widths:  
* Text: 55–60%  
* Screenshot: 40–45%  
Avoid two-column pages when:  
* Instructions require long numbered steps  
* Screenshot text becomes too small  
* Reading order becomes unclear  
* The page contains more than one table  
  
## 9. Callout styles  
Create reusable single-cell table styles or text-box styles.

Use for one key takeaway.  
## Avoid this  
**Label:** Avoid this **Fill:** Very light neutral or muted red tint **Border:** Muted red **Icon:** Optional stop or caution symbol

**Label:** Platform difference **Fill:** Light orange tint **Border:** Orange **Icon:** Device or comparison symbol

## Support  
**Label:** Get help **Fill:** Light gray **Border:** Primary blue

## 11. Screenshot placement  
## Standard image width  
Use:  
* Full-width screenshots: 6.5–6.8 inches  
* Half-width screenshots: 3–3.25 inches  
* Dialog screenshots: size for readability, not page filling  
## Image wrapping  
Recommended:  
* In Line with Text  
This supports predictable placement and accessible reading order.  
Use other wrapping only when tested carefully.  
## Image anchoring  
* Keep screenshots anchored to the related paragraph.  
* Select “Move with text.”  
* Avoid floating images that shift during edits.  
* Keep captions with screenshots.  
## Captions  
Use:  
**Figure 12. Set a custom follow-up date and reminder.**  
Do not include periods after short screenshot callouts inside the image unless required for clarity.  
  
## 12. Tables  
## Table design  
Use:  
* Header row in Midnight or Primary blue  
* White header text  
* Light row shading only when it improves scanning  
* Minimum 9.5–10 pt table text  
* Adequate cell padding  
* Repeating header rows across pages  
* No color-only status indicators  
## Table accessibility  
For every table:  
1. Identify the header row.  
2. Enable Repeat as Header Row.  
3. Add a short title or introduction.  
4. Avoid merged cells when possible.  
5. Do not use blank cells for visual spacing.  
6. Keep reading order left to right and top to bottom.  
7. Add text labels for status rather than color alone.  
## Prevent awkward splitting  
* Keep short tables together when possible.  
* Allow long rows to break only when readability remains clear.  
* Repeat column headings.  
* Move the entire table to the next page if only one row would remain.

## Document properties  
Set:  
* Title  
* Subject  
* Author or owner  
* Keywords  
* Comments or description  
Recommended title:  
**Take Control of Your Inbox: Classic Outlook for Windows**  
## Heading structure  
Use headings in logical order:  
* One document title  
* Heading 1 for major sections  
* Heading 2 for tasks  
* Heading 3 for subprocedures  
Do not skip from Heading 1 to Heading 3.  
## Alt text  
Add alt text to:  
* Screenshots  
* Workflow diagrams  
* Icons that convey meaning  
* Illustrations  
Mark decorative elements as decorative.  
## Links  
Use descriptive link text.  
## Lists  
Use Word list tools rather than typed symbols.  
## Color  
* Maintain sufficient contrast.  
* Do not rely only on color.  
* Keep body text dark on white.  
* Use icons or labels with colored callouts.  
## Reading order  
* Use inline screenshots.  
* Avoid complex floating layouts.  
* Test text boxes and callouts.  
* Use tables carefully.  
* Keep columns simple.  
## Accessibility Checker  
Run Word’s Accessibility Checker and resolve:  
* Missing alt text  
* Unclear link text  
* Table-header issues  
* Repeated blank characters  
* Reading-order concerns  
* Low contrast  
* Missing document title

## 19. Technical review process  
## Reviewers  
Recommended:  
* IT Training & Enablement  
* Outlook technical owner or Microsoft 365 administrator  
* Sales representative or associate  
* Shared mailbox owner  
* Copilot subject matter reviewer  
* Accessibility reviewer  
## Technical review questions  
For each procedure:  
1. Does the command exist in Classic Outlook?  
2. Does the menu path match the managed environment?  
3. Does the result match the description?  
4. Is the procedure safe for an employee to complete?  
5. Does the procedure require permissions not stated?  
6. Is administrator action being presented as a user action?  
7. Does the screenshot match the written steps?  
8. Are platform differences accurate?  
9. Are Copilot limitations correctly described?  
10. Is the support path correct?  
## Pilot test  
Ask two to four employees to complete selected tasks using only the guide.  
Recommended pilot tasks:  
* Flag a message and add a reminder  
* Create a category  
* Create a simple rule  
* Search for an attachment  
* Add or use a shared mailbox  
* Draft a reply with Copilot  
* Review a Copilot summary  
Record:  
* Completion success  
* Time needed  
* Confusing steps  
* Missing prerequisites  
* Screenshot usefulness  
* Searchability  
* Support questions

## 25. Review cadence  
Review:  
* At least every 12 months  
* When Microsoft changes Classic Outlook  
* When Copilot controls change  
* When shared mailbox behavior changes  
* When screenshots no longer match  
* When the Microsoft 365 Apps update channel changes  
* When support data shows repeated confusion  
* When a user reports outdated instructions  
* Before combining the guide into the master Outlook guide  
## Priority update triggers  
Update immediately when:  
* A procedure no longer works  
* A screenshot directs users to the wrong command  
* A step introduces security or privacy risk  
* A Copilot limitation changes  
* A support path changes  
* A shared mailbox permission workflow changes  
  
## 26. Retirement and replacement  
When a new approved version is published:  
1. Publish the replacement.  
2. Update SharePoint links.  
3. Update related resource pages.  
4. Mark the prior version as archived.  
5. Remove the prior version from active navigation.  
6. Preserve the Word source and PDF for history.  
7. Record the replacement URL.  
8. Update the version-control log.  
9. Confirm search promotes the current version.  
10. Do not delete the prior version unless governance approves an exception.
